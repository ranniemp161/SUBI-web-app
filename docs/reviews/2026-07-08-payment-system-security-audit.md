## Summary
The wallet app is the sole Stripe billing authority: users buy USD-micro credit bundles via Checkout or enable off-session auto-recharge, and the shared `@repo/db` ledger (append-only, CHECK-guarded cache) is spent by rough-cut. The core money invariants are, on the whole, correctly built: webhook signatures are cryptographically verified against the raw body, every grant is idempotent on a unique `stripe_event_id` / PaymentIntent id, credited amounts are derived server-side from allowlisted Stripe Prices (never from the client), overdraft is prevented by a DB CHECK rather than racy app-level reads, and the auto-recharge idempotency key is genuinely well-designed to make overlapping sweeps safe. The headline issues are resilience-and-test gaps rather than open exploits: the rate limiter and the ai-cut idempotency lock both **fail open** on a KV/Redis outage (removing the only double-charge guard on the ai-cut retry path), the cleanup cron **fails open** when `CRON_SECRET` is unset, and the most security-critical logic (real signature rejection, webhook idempotency at the DB level, and the anti-double-charge idempotency key) is mocked away in tests rather than exercised. No unauthenticated credit-grant path and no price/amount-tampering vector was found.

## Major

### Rate limiter and idempotency lock fail OPEN on a KV outage, on money-moving endpoints, `apps/wallet/src/lib/rate-limit.ts:66` and `apps/rough-cut/src/app/api/projects/[id]/ai-cut/route.ts:56`
**Problem**: `rateLimit()` returns `{ allowed: true }` both when KV is unconfigured (dev) and, more importantly, whenever the Redis call throws (`catch (error) { … failing open }`, rate-limit.ts:66-69). The ai-cut route reuses this same limiter as a distributed idempotency lock (`rateLimit(idempotency:<key>, 1, 86400)`, ai-cut/route.ts:59). `chargeAiCut`/`refundAiCut` are not themselves idempotent (they append an unkeyed ledger row every call — credits.ts:359, 393).
**Attack scenario / impact**: During a Redis degradation (or any transient Upstash error), (a) the per-user caps on `checkout`, `setup-intent`, `autorecharge` PATCH, and `ai-cut` all evaporate, letting an authenticated member create unbounded Stripe objects / trigger repeated off-session behaviour; and (b) the ai-cut `Idempotency-Key` 409 guard evaporates, so a client that retries a request whose response was lost is charged again for the same run with no dedup. The window is bounded (auth-gated members, KV outages are transient), but the failure mode on a payment system is silent double-charge / cost abuse exactly when the platform is already unhealthy.
**Suggested fix**: For the money/lock paths, fail *closed* (or degrade to a stricter fallback) rather than open — at minimum treat the idempotency-lock acquisition failure as "cannot prove first-run → refuse" for ai-cut, and consider a separate stricter policy for checkout/setup-intent than for read endpoints. Make `chargeAiCut`/`refundAiCut` idempotent on a caller-supplied key so the lock is defense-in-depth, not the sole guard.

### Security-critical behaviour is mocked, not tested (signature rejection, DB-level idempotency, the anti-double-charge key), `apps/wallet/src/app/api/webhooks/stripe/route.test.ts:20` and `apps/wallet/src/app/api/cron/autorecharge/route.test.ts:31`
**Problem**: The webhook test mocks `getStripe().webhooks.constructEvent` entirely; the "400 on an invalid signature" case just makes the mock throw (route.test.ts:136-143). Real signature verification against a forged/replayed body is never exercised. Idempotency (`depositPurchase`/`depositAutoRecharge` returning false on a duplicate `stripe_event_id`) is asserted only against a mocked `db.execute` that returns whatever `state.rows` is set to — the `ON CONFLICT … DO NOTHING` behaviour itself is never tested. The cron test mocks `autoRechargeIdempotencyKey` to the constant `"idem-key"` (cron/route.test.ts:31), so the single most important double-charge protection — that the key advances only on a real state change — is not integration-tested at the route level.
**Attack scenario / impact**: A regression that (for example) stopped passing the raw body to `constructEvent`, weakened the `ON CONFLICT` clause, or broke the idempotency-key derivation would pass the entire suite green. On a billing system these are the exact defects that cause forged-webhook credit grants or double charges.
**Suggested fix**: Add a test that feeds a real (test-mode) signed payload through the actual Stripe library and asserts a tampered body/secret is rejected; add a ledger-level idempotency test that exercises the real `ON CONFLICT` (or a Postgres-backed integration test); and let `autoRechargeIdempotencyKey` run unmocked in at least one cron test asserting a re-run reuses the same key and a post-success run does not.

## Minor

### Cleanup cron fails open when `CRON_SECRET` is unset, `apps/wallet/src/app/api/cron/cleanup/route.ts:10`
The guard is `if (process.env.CRON_SECRET && authHeader !== …)` — if the secret is missing the route is world-callable, and a test enshrines this (`cleanup/route.test.ts:58` "allows access … if CRON_SECRET is not set"). The autorecharge cron correctly fails *closed* (`!process.env.CRON_SECRET || …`, autorecharge/route.ts:60-63). Today cleanup is a no-op so impact is nil, but the inconsistent pattern means any future deletion logic added here would ship unauthenticated. Make it fail closed to match the autorecharge cron.

### Saved customer/payment-method can be bound to mismatched ids under concurrent first-time checkout, `apps/wallet/src/app/api/webhooks/stripe/route.ts:22` (`persistSavedCard`)
`setStripeCustomerId` is COALESCE (first-write-wins, autorecharge.ts:168) but `setDefaultPaymentMethod` always overwrites (autorecharge.ts:179). If a first-time user runs two checkouts concurrently, each with `customer_creation: "always"`, the stored `stripe_customer_id` may end up being customer A while the last-written `default_payment_method_id` is a PM attached to customer B. A later off-session auto-recharge then charges customer A with B's payment method and Stripe rejects it ("payment method does not belong to customer") — degrading to a decline / auto-disable, not a money loss, but a real dead-card foot-gun. Consider deriving/validating the PM's customer before persisting, or persisting both from the same session atomically.

### Auto-recharge amount is not constrained to whole cents, contradicting the rounding safety comment, `apps/wallet/src/lib/stripe.ts:140`
`microsToStripeMinorUnit` rounds `micros/10_000` and its comment claims "Amounts are validated to be whole-cent multiples of the retail bundles." That is true for bundles but **not** for auto-recharge, where `amountMicros` is an arbitrary client-supplied positive integer (autorecharge route validates only `.int().positive()`, `> threshold`, `<= MAX`). A crafted `amountMicros` just below a half-cent boundary is credited in full while the customer is charged the rounded-down cent, a sub-cent gain per recharge (capped at 3/day, so economically negligible, but the stated invariant is false). Either enforce whole-cent amounts server-side or credit exactly `microsToStripeMinorUnit(...) * 10_000` rather than the raw metadata micros.

### `chargeAiCut` / `refundAiCut` are not idempotent, `apps/rough-cut/src/lib/credits.ts:359,393`
Both append an unkeyed ledger row and mutate the balance on every call, unlike every other mutation here which is keyed (`stripe_event_id`, `month_key`, or the hold gate). The current caller (ai-cut/route.ts) always pairs a charge with at most one refund per request, so it is not exploitable today, but there is no structural guard: a future caller, a duplicated refund path, or the KV-lock outage above turns this into free-credit minting. Add a per-attempt idempotency key or a project/attempt uniqueness guard.

### Webhook `payment_intent.succeeded` credits `metadata.amountMicros`, not the amount actually captured, `apps/wallet/src/app/api/webhooks/stripe/route.ts:157`
The credit is taken from PI metadata rather than `pi.amount_received`. Metadata is written server-side in `chargeAutoRechargeOffSession` and the amounts are consistent today, so this is not currently exploitable, but sourcing the credited value from the confirmed captured amount would make the two impossible to drift apart (defense-in-depth, consistent with the "amounts come from Stripe" principle).

### Very chatty webhook logging of user ids / metadata, `apps/wallet/src/app/api/webhooks/stripe/route.ts:100,113,116,126,131`
Multiple `console.log` lines echo `session.id`, internal `userId` (UUID), and parsed `creditMicros` on every delivery. No secrets or PANs (good), but internal identifiers in logs are noise and mild information exposure; consider dropping to debug level or removing before compliance sign-off.

## Nits
- `apps/wallet/src/lib/stripe.ts:23`, `new Stripe(key)` pins no `apiVersion` — relies on the account default, which can silently change object shapes across Stripe upgrades.
- `apps/wallet/src/proxy.ts:9-16`, `"/api/cron(.*)"` and `"/api/billing/bundles(.*)"` are broad public matchers — any future sub-route added under these paths is public-by-default and must remember to self-gate.
- `apps/wallet/src/app/api/billing/checkout/route.ts:53`, the form-encoded (non-preflighted) POST path is a CSRF surface; impact is negligible (a victim-initiated checkout credits the victim's own account, no charge without card entry), but the JSON path is the only one that needs to exist.
- `apps/wallet/src/app/dashboard/transaction-history.tsx:26-27`, `REASON_LABELS` lists `hold`/`settle`/`reclaim` which are not members of the `credit_ledger_reason` enum — dead labels.
- `apps/rough-cut/src/lib/credits.test.ts:209,222,247`, several ledger mocks return `{ tokens: … }` rows though the code reads `balance_micros` — the assertions pass regardless, so the mock shape is misleading.

## Strengths
- Webhook signature verification is real: `getStripe().webhooks.constructEvent(rawBody, signature, secret)` over `request.text()` (not `JSON.parse`), with a missing-secret hard 500 and a missing/invalid-signature 400 before any DB work (webhooks/stripe/route.ts:86-96).
- Idempotency is enforced at the database, not in memory: every deposit uses `INSERT … ON CONFLICT (stripe_event_id) DO NOTHING` and returns false on a duplicate (credits.ts:331, autorecharge.ts:127), backed by the unique constraint in schema.ts:137.
- Credited amounts are server-derived from the allowlisted Stripe Price's `metadata.credit_micros` and written into session metadata at creation time; the webhook only trusts what it wrote (checkout/route.ts:96, stripe.ts:51). No client dollar amount is ever trusted.
- Overdraft safety rests on the `users_balance_micros_nonneg` CHECK with single-statement CTE mutations, not on racy read-then-write (schema.ts:65, credits.ts commentary) — concurrent spends serialize correctly.
- The auto-recharge idempotency key advances only on real state change (`s<successes>:f<failures>`), which correctly makes overlapping sweeps dedup at Stripe while still freeing a fresh key after a success or decline (autorecharge.ts:109-115) — a genuinely careful piece of double-charge design. Failure accounting is centralised in the synchronous sweep with the webhook `payment_intent.*` handlers as idempotent backstops, avoiding double-counting.
- The autorecharge cron fails closed on a missing `CRON_SECRET`, and per-user resources are always scoped to the server-resolved `user.id` from `getAuthorizedDbUser(clerkId)` — no client-supplied user/customer id is trusted (no IDOR found on the billing routes).

## Test coverage
Breadth is good: every billing route has happy-path + auth (401/403) + rate-limit (429) + malformed-body coverage, the cron sweep covers charge/decline/cap/error branches, and the ledger helpers cover the CHECK-violation and duplicate-delivery mappings. The gaps are in the highest-risk logic (see Major #2): real signature rejection is never exercised (constructEvent is fully mocked), DB-level idempotency is asserted against a mock that can't reproduce `ON CONFLICT`, and the anti-double-charge idempotency key is mocked to a constant in the cron test — so the three defects most likely to cause forged grants or double charges would not be caught. The KV fail-open behaviour on the money paths is also untested. Recommend adding: a real signed-payload webhook test, a Postgres-backed idempotency/CHECK integration test, an unmocked idempotency-key cron test, and a fail-closed assertion for the lock/limiter on the ai-cut retry path.
