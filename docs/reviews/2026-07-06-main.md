## Summary
This change introduces the wallet and billing infrastructure, including Stripe checkout and webhook handling, and credit ledger logic. While the database atomicity and webhook idempotency are exceptionally well thought out, there are critical integration bugs in the checkout flow and route protection that will completely block users from viewing or buying bundles. Additionally, the internal cost reporting logic computes inverted values for refunds and settlements.

## Blockers
### 🔴 Public bundles API is blocked by Clerk middleware, `apps/wallet/src/proxy.ts:3`
**Problem**: The `/api/billing/bundles` route is designed to be public and edge-cacheable, but `isPublicRoute` in `proxy.ts` does not include it.
**Why it matters**: `auth.protect()` will intercept unauthenticated requests and return a 401 Unauthorized, preventing the client from fetching the bundles before login or on public pages.
**Suggested fix**: Add `/api/billing/bundles(.*)` to the `isPublicRoute` matcher in `proxy.ts`.

### 🔴 Checkout form cannot be submitted and API expects wrong content type, `apps/wallet/src/app/dashboard/page.tsx:51`
**Problem**: In the dashboard, the bundle selection uses `<button type="button">` inside a standard HTML `<form>`, which won't trigger submission. Furthermore, if submitted, the form sends `application/x-www-form-urlencoded` data, but the API route at `apps/wallet/src/app/api/billing/checkout/route.ts` strictly expects and parses JSON (`await request.json()`).
**Why it matters**: Users will not be able to initiate a checkout. Clicking the button will do nothing, and if forced, it will result in a 400 "Invalid request body" error.
**Suggested fix**: Change the button to `type="submit"`, and either update the API route to handle `request.formData()`, or replace the HTML form in the UI with an `onClick` handler that calls `fetch` with a JSON body.

## Major
### 🟠 Cost micros calculation has reversed signs for refunds and settlements, `apps/wallet/src/lib/credits.ts:257`
**Problem**: In `settleHold` and `refundAiCut`, the `cost_micros` calculation uses the wrong sign. In `settleHold`, `delta * TRANSCRIPTION_COST_MICROS_PER_SECOND` means a negative delta (charging a shortfall) produces a negative cost, and a positive delta (refunding) produces a positive cost. Similarly, `refundAiCut` uses `costSeconds * COST` (positive) for a refund.
**Why it matters**: The `credit_ledger` will have wildly inaccurate cost aggregates, breaking internal margin visibility and financial reporting by decreasing costs when charging users and increasing costs when refunding them.
**Suggested fix**: Use `-delta * TRANSCRIPTION_COST_MICROS_PER_SECOND` in `settleHold` and `-costSeconds * AI_CUT_COST_MICROS_PER_SECOND` in `refundAiCut` and `reclaimStaleHold`.

### 🟠 getBundles fails entirely if a single Stripe price is deleted, `apps/wallet/src/lib/stripe.ts:61`
**Problem**: `getBundles()` uses `Promise.all` with `stripe.prices.retrieve` across all configured `STRIPE_PRICE_IDS`. If any price in the allowlist is deleted or invalid in Stripe, the entire promise rejects.
**Why it matters**: A single misconfigured ID in the environment variable or a deleted product in the Stripe dashboard will cause a 500 error, taking down the entire dashboard and billing API instead of just omitting the broken bundle.
**Suggested fix**: Use `Promise.allSettled` or catch individual rejections within the `map` loop, logging the error and filtering out the failed retrievals before returning the active bundles.

## Minor
### 🟡 Ledger history is capped at 50 with no pagination, `apps/wallet/src/app/dashboard/page.tsx:28`
**Problem**: The ledger query uses `.limit(50)` but the UI provides no pagination or "load more" mechanism.
**Why it matters**: Active users will quickly exceed 50 transactions and lose visibility into their earlier billing history.
**Suggested fix**: Implement cursor-based pagination or an offset parameter to allow users to view older ledger entries.

### 🟡 Confusing variable name and misleading error log in depositPurchase, `apps/wallet/src/lib/credits.ts:287`
**Problem**: `depositPurchase` receives `session.id` as the `stripeEventId` parameter. While functionally correct for ensuring idempotency per checkout session, the naming is misleading. Additionally, if the `UPDATE users` statement matches 0 rows (e.g., if the user was deleted), it returns `false`, which the webhook logs as a "Duplicate Stripe webhook delivery".
**Why it matters**: It makes future maintenance confusing and obscures true errors (missing user) by mislabeling them as benign duplicates in the logs.
**Suggested fix**: Rename `stripeEventId` to `stripeSessionId` or `idempotencyKey` in the function and DB schema. Adjust the SQL query or return type to distinguish between an `ON CONFLICT` duplicate and a missing user row.

## Strengths
- Excellent use of atomic CTE pipelines in `credits.ts` to ensure consistency and prevent overdrafts without complex transactions or locks.
- Idempotency in the webhook handler ensures that racing or retried Stripe deliveries cannot double-credit a user.
- Clever use of SQL check constraints (`users_tokens_nonneg`) as the definitive concurrency gate.
