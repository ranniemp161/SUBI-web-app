## Major
### 🟠 Baseline instructions point to a "hash" that isn't in `_journal.json`, `packages/db/MIGRATIONS.md:47-49`
**Problem**: Step 2 of the "First-deploy baseline" section says to insert a row for `0000` "using its hash + `when` timestamp from `drizzle/meta/_journal.json`". But `_journal.json` entries only contain `idx`, `version`, `when`, `tag`, `breakpoints` (confirmed in `packages/db/drizzle/meta/_journal.json:5-11`) — there is no `hash` field. Drizzle's migrator (`node_modules/drizzle-orm/migrator.cjs:56`) computes `hash` at runtime as `sha256(fileContents)` of the migration `.sql` file itself, not something stored in the journal.
**Why it matters**: Whoever executes the baseline (a manual, one-time, prod-only procedure per the runbook) will look in `_journal.json` for a hash, not find one, and get stuck or guess. It happens to be low-risk functionally — `neon-http/migrator.cjs:45` gates skip/apply purely on the `created_at` timestamp comparison, not on the hash value, so any placeholder string would technically "work" for the NOT NULL column — but the runbook shouldn't rely on that undocumented behavior, and an operator following the doc literally has no path to get "the hash" as described.
**Suggested fix**: Correct the instruction to either (a) compute the hash directly — e.g. `sha256sum drizzle/0000_thick_young_avengers.sql` (or the Node `crypto.createHash('sha256')` equivalent) — and use `when` from the journal for `created_at`, or (b) explicitly note that the hash value is unused for gating (only `created_at` matters) so any placeholder is acceptable, if that's confirmed intentional.
