# Child 2 — Reselect Gated Processing

## Summary

This child makes reselecting the source video the real trigger for the studio's automatic processing chain. Today the chain (mechanical cut, then AI polish) fires the moment the transcript alone is ready, because the mechanical cut only needs the transcript's words, not the video file. Going forward it must not fire until the user has reselected their source video and passed the existing duration verification, in addition to the two conditions it already checks. The moment reselect succeeds, a full-screen loading state with the exact copy "A.I. is doing the rough cut in the background..." spans both phases. Failure handling, the manual retry button, and every legacy project behavior stay exactly as they are today.

**Covers**: AC-6, AC-7, AC-8, AC-9, AC-10, AC-11, AC-12.

## Revision (2026-07-14) — AC-12 added

The first build landed AC-9 by reusing `AiCutOverlay` exactly as it existed: an overlay scoped to the video-preview container only. Visually this meant the transcript panel, timeline, and rail were already visible and interactive behind the loader the instant reselect succeeded — not the full-page takeover "full-screen loading state" was meant to describe. The engineer clarified the actual intent after seeing the shipped behavior: the studio's real editing chrome must not render at all until the whole chain (mechanical + AI) has settled — success or failure — so the user never sees a partially-loaded editor mid-cut. AC-12 below is the added, independently-checkable requirement; AC-9's loader copy requirement is unchanged, it's now delivered by a page-level loading state instead of a container-scoped overlay. On a failure or 402 during this wait, the page still redirects into the normal editor showing the mechanical result — AC-10 is otherwise completely unaffected.

## Context

The studio page is `apps/rough-cut/src/app/(app)/dashboard/[id]/page.tsx` (route `/dashboard/[id]`). It fetches the project from `GET /api/projects/${id}` on mount (fetch effect at lines 218-276). That effect computes the "fresh" condition into `freshOnLoadRef` (lines 260-261): `data.transcriptStatus === "ready" && !savedEdl`, where `savedEdl` discards a saved edit list that keeps nothing as corrupt (lines 240-241).

The reselected video lives only in client state: `sourceFile` (line 141), set by the existing reselect prompt. When `sourceUrl` is null (no file reselected yet), the page renders a "STEP 2 reselect your video file" screen (lines 1199-1222) containing `FilePicker` with its duration-verification props (`expectedDurationMs`, `expectedFileSize`, `expectedFileName`, `expectedFileType`, lines 1211-1217); `onFileSelected` simply calls `setSourceFile(file)`. The duration check itself (1500ms tolerance, block with a clear message on mismatch) lives inside `FilePicker` (`apps/rough-cut/src/components/file-picker.tsx`) and was shipped by ADR 0001 child 3 (`docs/adr/rough-cut/0001-editor-studio-ux-safety/0003-video-reselect-verification.md`); this child does not touch it.

The auto-chain is `runAutoChain` (lines 839-863): it builds the mechanical cut via `buildInitialEDL` (pure, client-side, transcript words only), applies it, and when `project?.aiPolishRequested && project.aiCutRuns.length === 0` chains into `runAiCut` with the freshly built edit list. The firing effect (lines 871-886) guards on `autoChainedRef` (at most once per mounted studio), `freshOnLoadRef` (the "fresh" condition), a ready transcript, `edl === null`, and non-empty words/duration. Note what is absent: nothing in that effect reads `sourceFile`. The chain fires whether or not the user has reselected the video, which is exactly the behavior this child changes.

The loading visual is `AiCutOverlay` (lines 1582-1606), rendered while `aiBusy || autoCutBusy` (line 1364). Its current copy is "AI is reviewing your transcript…" with the subline "Finding false starts, stumbles & flubbed takes". `autoCutBusy` (line 150) spans the whole chain; `aiBusy` (line 738) spans the AI phase, whether triggered automatically or by the manual "Polish with AI" button (visibility keyed off `noAiRunYet`, line 1241, which is `(project?.aiCutRuns.length ?? 0) === 0`). Failure and 402 handling live inside `runAiCut` and the server route (`POST /api/projects/[id]/ai-cut`: atomic claim via `claimAiCutSlot` in `apps/rough-cut/src/lib/projects.ts`, charge via `chargeAiCut` in `apps/rough-cut/src/lib/credits.ts`, Gemini via `runAiRoughCut` in `apps/rough-cut/src/lib/ai-rough-cut.ts`, refund on failure via `refundAiCutQuietly`).

One subtlety worth naming: because the reselect prompt currently renders whenever `sourceUrl` is null, a fresh project's first visible state today is already the reselect screen, while the auto-chain silently runs behind or before it. So AC-7 (reselect prompt first) mostly already holds visually; the substantive change is AC-8 (the gate) and AC-9 (the loader appearing at the moment reselect succeeds, with new copy).

## Requirements

Covers: AC-6, AC-7, AC-8, AC-9, AC-10, AC-11, AC-12.

**AC-12** (added 2026-07-14): The moment reselect succeeds, the studio page renders ONLY a full-page loading state — no transcript panel, no timeline, no rail, no editor chrome of any kind, just the loader with AC-9's exact copy — instead of today's editor-chrome-plus-scoped-overlay. The page swaps to the normal editor view only once the auto-chain has fully settled: on success (both phases done), on an AI failure, or on a 402. In every one of those settle cases the user lands in the editor exactly as AC-10 already specifies (mechanical result applied, existing toast/add-funds link, manual "Polish with AI" retry visible) — AC-12 only changes what's on screen *before* settling, never what happens after. **Same-day follow-up:** the full-page loading state also shows a linear progress bar with a live percentage (the engineer's explicit ask, for readability during a wait that can run to a couple of minutes) — same duration-calibrated estimate `AiCutOverlay` already used, just also driving a labeled bar under the message instead of only a ring inside the small overlay.

## Decision

**Chosen option**: (umbrella-level options analysis; this child is Option 1's concrete build for the studio side.)

**RECOMMEND 2 — new states inside the existing studio page, no new route.** The reselect prompt and the loading state are implemented as additional client states within the existing studio page component, reusing the existing `FilePicker` (with its duration-verification props unchanged) and the relabeled `AiCutOverlay`. Clicking a "Ready for step 2" row navigates to the same `/dashboard/[id]` route it does today. Runner-up considered: a separate route or page dedicated to step 2 (reselect, then process, then hand off to the editor). Rejected, per the engineer's explicit choice, because the studio page already owns the reselect prompt, the `sourceFile` state, and the auto-chain logic; splitting them across routes would force either duplicating that machinery or refetching and re-deriving project state on a second page, adding navigation and data-refetch complexity for no behavior the single page cannot already express.

**RECOMMEND 3 — gate the auto-chain itself on reselect, not just the loader.** `runAutoChain`'s firing effect gains a third clause: the reselected source file must be present (it can only be present by having passed `FilePicker`'s duration verification, since that is the only way `setSourceFile` is reached). Runner-up considered: leave the auto-chain decoupled from reselect as it is today and only gate the loading state's visibility on reselect, so the work happens early but the user only sees it after reselecting. Rejected because it would mean the mechanical cut, and possibly a real AI charge, could silently start in the background before the user ever reselected their video, which contradicts the engineer's explicit sequencing (the moment the video is reselected, processing starts) and would let money be spent without the user having taken the action they associate with starting it.

## Feature design

**Data model**: No schema change. The "reselected and verified" fact is session-local client state (`sourceFile`), never persisted; a user who reselects, closes the tab before the chain finishes, and returns simply reselects again, and the server-side claim machinery (unchanged) guarantees no double AI charge across those sessions.

**State transitions** (fresh project, this child's scope):
- Studio opens on a fresh project (ready, no usable saved edit list): first visible state is the existing reselect prompt (`sourceUrl === null` branch). Nothing processes yet.
- Reselect succeeds (file passes `FilePicker`'s duration verification, `setSourceFile` fires): the auto-chain's firing effect now has all its clauses satisfied and fires once; `autoCutBusy` goes true; the page renders ONLY the full-page loading state (AC-12) — the transcript panel, timeline, and rail do not mount yet.
- Mechanical phase completes and, because the project is fresh with `aiPolishRequested` true and zero runs, the AI phase runs immediately (`aiBusy`); the full-page loading state stays up, unchanged copy, spanning both phases.
- Both phases settle (success, failure, or 402): the full-page loading state unmounts and the normal editor (transcript panel, timeline, rail, all editor chrome) mounts for the first time — this is the only point at which the user ever sees the studio's editing surface for a fresh project.
- Reselect fails duration verification: `FilePicker` blocks with its existing message and lets the user retry; nothing fires, the reselect prompt stays up (not the loading state). Unchanged behavior, only its position in the flow is now load-bearing.

**API surface**: No route changes. `POST /api/projects/[id]/ai-cut` is called with the same inputs, auth, and error surface as today; only the client-side moment of the call moves.

**Key invariants**:
- **The gating change (the single highest-risk, most load-bearing change in this ADR).** The auto-chain firing effect (currently lines 871-886) gains exactly one new clause: `sourceFile !== null` (equivalently, `sourceFile` joins the effect's dependency array and a null check joins its early returns). The full firing condition becomes: not already chained (`autoChainedRef`), fresh on load (`freshOnLoadRef`), project ready, `edl === null`, words and duration present, AND source file reselected. Getting this wrong in either direction is the regression this ADR exists to prevent: too permissive (clause missing or misplaced) and the chain fires on transcript-readiness alone again, spending money before the user acts; too strict (e.g. accidentally requiring `sourceFile` on non-fresh projects too) and either the chain never fires or legacy projects get blocked from the editor. The clause must live in the firing effect, not inside `runAutoChain` itself, so the effect naturally re-evaluates when `sourceFile` lands; a check only inside `runAutoChain` would silently no-op once and never retry, because `autoChainedRef` would already be spent. `autoChainedRef` must therefore be set only when the chain actually fires, never on an early return.
- The AI phase still receives the freshly built mechanical edit list as an explicit argument (`runAiCut(mechanical)`, line 858), never a stale closed-over `edl`; this ADR 0003 invariant is untouched and must survive the edit.
- The loader's copy is exactly "A.I. is doing the rough cut in the background..." (verbatim, per the engineer) — unchanged from the first build.
- **AC-12, the full-page loading state (added 2026-07-14).** While `aiBusy || autoCutBusy` is true AND the project is on the fresh, gated path (i.e. the auto-chain either is running or is about to, per the AC-8 gate), the studio page renders ONLY the loading state — a dedicated full-page component, not the existing container-scoped `AiCutOverlay` — and does not mount the transcript panel, timeline, rail, or any other editor chrome. This matters specifically because those components currently mount unconditionally as soon as `sourceUrl` is non-null (i.e. the instant `setSourceFile` fires), before the chain has even started; AC-12 requires gating their mount on the chain having settled, not just on `sourceFile` being present. `AiCutOverlay` (the small, container-scoped component) keeps existing exactly as-is for its other caller — the manual "Polish with AI" button's on-demand run, which happens after the user is already in the editor and does not hide any chrome.
- Failure and 402 handling are byte-for-byte today's behavior: mechanical result stays applied and visible, existing error toast, 402 gets the existing add-funds deep link to `WALLET_DASHBOARD_URL`, the charge is refunded via `refundAiCutQuietly`, and the manual "Polish with AI ($X)" button (visible only while `noAiRunYet`, gone permanently after one success) remains the retry path. Explicitly required; do not remove it. AC-12 changes nothing about this — it only changes what renders *before* the chain settles, and the editor (with this exact failure treatment) is what renders the moment it does.
- Legacy inertness: a project with a saved edit list or an existing AI Cut run sets `freshOnLoadRef` false at load and is provably unreachable by the gate, so it opens straight into the editor exactly as today (its own reselect-for-playback prompt, when `sourceUrl` is null, is the same screen as today and gates nothing) — and, because the gate is unreachable, AC-12's full-page loading state never renders for it either; the editor chrome mounts immediately once `sourceUrl` is set, same as before this child.
- The duration verification itself (ADR 0001 child 3) is completely unchanged: same 1500ms tolerance, same block-and-retry on mismatch. This child changes only when the prompt appears in the flow and what a successful reselect now triggers.

**Security model**: Unchanged. No new endpoint, no new client-controlled input reaches the server; the claim, charge, idempotency, and refund guarantees of `POST /api/projects/[id]/ai-cut` are exactly as ADR 0002/0003 left them, and remain the authoritative guard against double charges regardless of anything the client sequencing does.

**Critical test scenarios**:
- **The regression this ADR must prevent**: open the studio on a fresh, ready project and do not reselect; assert the auto-chain never fires (no `buildInitialEDL` application, no call to `POST /api/projects/[id]/ai-cut`) even though the transcript alone would be enough to build the mechanical cut, verifies **AC-8**.
- Fresh project, reselect succeeds: the overlay appears with the exact copy "A.I. is doing the rough cut in the background...", the mechanical cut applies, the AI phase runs under the same overlay, and the user lands in the editor with one stored run, verifies **AC-7, AC-8, AC-9**.
- Reselect fails duration verification: the existing mismatch message shows, nothing fires, and a subsequent correct reselect triggers the chain normally, verifies **AC-7, AC-8**.
- AI phase fails (forced Gemini error) or 402s after reselect: mechanical result stays, existing toast (and for 402 the add-funds deep link) appears, charge refunded, manual "Polish with AI" button visible, verifies **AC-10**.
- Legacy project (saved edit list, or an existing AI Cut run): studio opens straight into the editor as today, no gate, no auto-fire, no relabeled behavior, and no full-page loading state ever renders for it, verifies **AC-11, AC-12**.
- Navigation: a "Ready for step 2" dashboard row links to `/dashboard/[id]` and nowhere else, verifies **AC-6**.
- **Full-page loading state (AC-12)**: on a fresh project, after reselect, assert the transcript panel, timeline, and rail are NOT present while the chain is running (only the loading state's copy is), and assert they DO appear once the chain settles (success case) — and separately, that they also appear (with the mechanical result + retry button) once the chain settles via an AI failure/402, verifies **AC-12**.

## Consequences

**Positive**: processing and spending are now tied to a visible user action (reselecting the video), so nothing cuts or charges off-screen before the user has resumed the project. The user gets one honest loading state, with copy that matches what is actually happening, spanning the whole automatic chain.

**Negative / tradeoffs**: the mechanical cut now waits for reselect even though it technically never needed the video file, so a user who was previously handed an already-cut timeline the moment they opened a fresh project now waits through the chain after reselecting instead. The auto-chain effect gains one more clause whose exact placement is load-bearing (see Key invariants); this is the concentrated risk of the whole ADR and is why it has a dedicated build task and a dedicated regression test. **(Added 2026-07-14)** AC-12 means a fresh project's editor chrome (transcript panel, timeline, rail) now mounts later than it did in the first build of this ADR — only once the chain settles — so there is nothing to interact with (no early peek at the transcript, no early scrubbing) during what can be a multi-second-to-low-minutes wait for the AI phase; this is deliberate, not a regression, per the engineer's explicit call that a partially-loaded editor mid-cut is worse than a clean wait.

**Neutral**: `FilePicker`, its duration verification, `runAiCut`'s failure branches, `refundAiCutQuietly`, the manual "Polish with AI" button, and the beforeunload warning while the AI pass runs are all reused exactly as they exist today; this child adds no new component and no new route.
