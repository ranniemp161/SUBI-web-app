# Hardening, main, 2026-07-08

**Analysed by**: systems level review on Gemini 1.5 Flash
**Scope**: 44 files, main compared to origin/main
**Risk posture**: Harden before merge

## Summary
The wallet and auto recharge changes introduce critical production interfaces (specifically off session billing, public Stripe webhooks, and cron sweeper endpoints). This review focuses on preventing unauthorized runs, performance limits under load, and rate limit protections.

## Must-fix before merge
### 🔴 Falsy cron secret authorization bypass, `apps/wallet/src/app/api/cron/autorecharge/route.ts:60`
**Scenario**: If the `CRON_SECRET` environment variable is not defined or is an empty string in production, the condition `process.env.CRON_SECRET && authHeader !== ...` evaluates to false, which completely bypasses the token check. Anyone can then trigger the endpoint.
**Impact**: High. Unauthorized actors could spam the sweeper endpoint, trigger unwanted off session card charges for users, and cause major billing disputes.
**Mitigation**: Force the secret verification to be strict. Fail closed if the secret is missing in production.
**Verify with**: Run automated unit tests in `route.test.ts` with an empty or missing `CRON_SECRET` and confirm it returns a 401 response.

## Should-harden
### 🟠 Sequential candidates sweeper loop times out under load, `apps/wallet/src/app/api/cron/autorecharge/route.ts:74`
**Scenario**: The sweeper processes all eligible users sequentially in a single loop. Each user charge requires several database checks, a network call to Stripe, and a notification send. As user numbers grow, this sequence will easily exceed the Next.js serverless request timeout (configured at 300 seconds).
**Impact**: Medium. Later users in the list will never get topped up, causing unexpected service blocks when their balances reach zero.
**Mitigation**: Implement batch processing (using parallel chunks with concurrency control) or offload candidate tasks to a message queue.
**Verify with**: Simulate 200 candidates in local test runs and monitor overall processing time.

## Watch / accept
* 🟡 `apps/wallet/src/lib/autorecharge.ts:59`, Sweep race condition with concurrent manual deposit. A user selected by the sweep because their balance is below the threshold could still be charged off session even if they manually deposit funds in the time window before the Stripe charge executes. This is acceptable because the balance is credited correctly to their account, and the maximum successful charges per day cap prevents runaway charges.
* 🟡 `apps/wallet/src/app/api/webhooks/stripe/route.ts:145`, Card saving failures are non blocking. If saving the payment method or Stripe customer ID fails after a successful purchase, the error is caught and logged but the webhook returns a 200 response. This is acceptable because card saving issues should never block or revert a completed credit purchase.

## Already covered
* **Idempotency on Stripe ID**: Every balance deposit (manual purchase and auto recharge) is recorded in the append only ledger with a unique Stripe transaction ID constraint. Duplicate webhook calls are safely ignored without double crediting.
* **Daily recharge limits**: The auto recharge sweep counts the number of successful recharges for the user in the past 24 hours and stops immediately if it exceeds the limit.
* **Webhook signature verification**: Every webhook request is verified using the secret signing key before processing the body.
