# Hardening, main, 2026-07-08

**Analysed by**: systems level review on Gemini 1.5 Flash
**Scope**: 44 files, main compared to origin/main
**Risk posture**: Harden before merge

## Summary
The wallet and auto recharge changes introduce critical production interfaces (specifically off session billing, public Stripe webhooks, and cron sweeper endpoints). This review focuses on preventing unauthorized runs, performance limits under load, and rate limit protections.

## Must-fix before merge
### 🔴 Falsy cron secret authorization bypass, `apps/wallet/src/app/api/cron/autorecharge/route.ts:60`
**Scenario**: If the `CRON_SECRET` environment variable is not defined or is an empty string in production, the condition `process.env.CRON_SECRET && authHeader !== ...` evaluates to false, which completely bypasses the token check. Anyone can then trigger the endpoint.
**Impact**: High. Unauthorized actors could spam the sweeper endpoint, trigger unwanted off session card charges for users, and cause major billing disputes.
**Mitigation**: Force the secret verification to be strict. Fail closed if the secret is missing in production.
**Verify with**: Run automated unit tests in `route.test.ts` with an empty or missing `CRON_SECRET` and confirm it returns a 401 response.

## Should-harden
### 🟠 Sequential candidates sweeper loop times out under load, `apps/wallet/src/app/api/cron/autorecharge/route.ts:74`
**Scenario**: The sweeper processes all eligible users sequentially in a single loop. Each user charge requires several database checks, a network call to Stripe, and a notification send. As user numbers grow, this sequence will easily exceed the Next.js serverless request timeout (configured at 300 seconds).
**Impact**: Medium. Later users in the list will never get topped up, causing unexpected service blocks when their balances reach zero.
**Mitigation**: Implement batch processing (using parallel chunks with concurrency control) or offload candidate tasks to a message queue.
**Verify with**: Simulate 200 candidates in local test runs and monitor overall processing time.

## Watch / accept
* 🟡 `apps/wallet/src/lib/autorecharge.ts:59`, Sweep race condition with concurrent manual deposit. A user selected by the sweep because their balance is below the threshold could still be charged off session even if they manually deposit funds in the time window before the Stripe charge executes. This is acceptable because the balance is credited correctly to their account, and the maximum successful charges per day cap prevents runaway charges.
* 🟡 `apps/wallet/src/app/api/webhooks/stripe/route.ts:145`, Card saving failures are non blocking. If saving the payment method or Stripe customer ID fails after a successful purchase, the error is caught and logged but the webhook returns a 200 response. This is acceptable because card saving issues should never block or revert a completed credit purchase.

## Already covered
* **Idempotency on Stripe ID**: Every balance deposit (manual purchase and auto recharge) is recorded in the append only ledger with a unique Stripe transaction ID constraint. Duplicate webhook calls are safely ignored without double crediting.
* **Daily recharge limits**: The auto recharge sweep counts the number of successful recharges for the user in the past 24 hours and stops immediately if it exceeds the limit.
* **Webhook signature verification**: Every webhook request is verified using the secret signing key before processing the body.

# Hardening, main (add-card client form), 2026-07-08

**Analysed by**: systems-level review on Claude Sonnet 5
**Scope**: 5 files (+ 1 lockfile skipped), origin/main..HEAD (1 unpushed commit)
**Risk posture**: ~~Harden before merge~~ → **Ship as-is** (all 3 should-harden items fixed same day, 2026-07-08)

## Summary
This change wires a real client-side Stripe Elements card-capture form into the previously-stubbed "Add card" button. The backend (SetupIntent creation, webhook persistence) predates this change and is out of scope. The new risk was concentrated in three places: a client confirm call that could leave the UI permanently stuck on an unhandled rejection, a success message that raced the webhook that actually persists the card, and a missing publishable-key guard that failed silently rather than loudly in an app that has not been deployed yet. None of this could ever double-charge money (SetupIntents are zero-dollar auth). All three were fixed, tested, and the fixes were typechecked/linted/built clean the same day — see each item below.

## Should-harden

_All three items below were fixed 2026-07-08, same day as this review. Kept here (marked fixed) rather than deleted, so the checklist stays an honest record of what was found and closed._

### ✅ Unhandled rejection leaves the form stuck mid-submit — FIXED, `apps/wallet/src/app/dashboard/add-card-form.tsx`
**Scenario**: `handleSubmit` calls `await stripe.confirmSetup(...)` with no try/catch. Stripe.js normally resolves with `{ error }` rather than throwing, but a network drop, an ad blocker on `m.stripe.network`, or a Stripe.js internal exception can still reject the promise. When that happens, `setSubmitting(false)` in the success/error paths never runs because execution never reaches past the `await`.
**Impact**: The "Save card" button stays disabled and shows "Saving..." forever with no error message and no way to retry except reloading the page. On a billing form that reads as "the app is broken" right when a user is trying to hand a company their card.
**Fix applied**: wrapped the `confirmSetup` call in try/catch/finally — `setSubmitting(false)` now always runs, and a caught rejection shows "Something went wrong saving the card. Please try again."
**Verified with**: new `apps/wallet/src/app/dashboard/add-card-form.test.tsx` — `mockConfirmSetup.mockRejectedValueOnce(...)`, asserts the button re-enables and the error renders. Passing.

### ✅ "Card saved" message and refresh race the persisting webhook — FIXED, `apps/wallet/src/app/dashboard/autorecharge-panel.tsx`
**Scenario**: `handleCardSaved` fires as soon as the client-side `stripe.confirmSetup` call reports `setupIntent.status === "succeeded"`. The actual DB write (`setDefaultPaymentMethod`, invoked from `apps/wallet/src/app/api/webhooks/stripe/route.ts` on the `setup_intent.succeeded` event) happens on a separate, asynchronous Stripe webhook delivery that is not guaranteed to have landed by the time the client sees "succeeded." `router.refresh()` re-fetches the server component's props immediately afterward.
**Impact**: Under normal webhook latency this resolves in well under a second, but Stripe makes no latency guarantee. In that window the user could see "Card saved." while the refreshed panel still renders "No card saved."
**Fix applied**: `handleCardSaved` now shows "Card confirmed, finishing setup…" first, then polls `GET /api/billing/autorecharge` (up to 5 attempts, 400ms apart) for `hasCard: true` before declaring "Card saved." If the webhook still hasn't landed after the poll budget, it shows an honest "Card confirmed. It may take a moment to appear below." instead of a message that could contradict the visible UI.
**Verified with**: two new tests in `autorecharge-panel.test.tsx` — one where the poll succeeds immediately ("Card saved."), one where it never persists within the budget (softened message). Both passing. Also click-tested live in the browser by the user 2026-07-08 (see roadmap).

### ✅ Silent breakage when `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` is unset — FIXED, `apps/wallet/src/lib/stripe-client.ts`
**Scenario**: `getStripeClient()` fell back to `loadStripe(process.env.NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY ?? "")` with no presence check. This app has not been deployed yet and the var wasn't on any deploy checklist.
**Impact**: If the var were missing in a prod/preview deploy, the card form would be permanently dead with no error anywhere — the kind of failure a manual smoke test misses.
**Fix applied**: `getStripeClient()` now throws a clear error if the key is missing. `AddCardForm` catches that at render time and shows an inline "Billing is misconfigured (Stripe key missing). Card setup is unavailable right now." instead of crashing the page or silently failing.
**Verified with**: new test in `add-card-form.test.tsx` — mocks `getStripeClient` to throw, asserts the inline error renders instead of the form. Passing. Still recommend adding this var to whatever checklist gates the first production deploy — the code now fails loudly, but the var should still be provisioned before go-live.

## Watch / accept
- 🟡 `apps/wallet/src/app/dashboard/autorecharge-panel.tsx:338-354`, the "Add card" / "Replace card" trigger button stays enabled while the form is already open (`setupClientSecret` set). Clicking it again fetches a second `SetupIntent` and replaces `setupClientSecret`, but `AddCardForm`'s `<Elements>` is not remounted (no `key` on the component), so `@stripe/react-stripe-js` will likely keep confirming against the first SetupIntent's Elements instance rather than picking up the new client secret -- react-stripe-js does not support swapping `clientSecret` on a live `Elements` instance. Acceptable to watch rather than fix now because a stray extra SetupIntent create is free (no charge), bounded by the existing 10-per-hour rate limit, and the worst case is a confusing double form rather than a wrong charge; the cheap fix later is to hide/disable the trigger while the form is open and add `key={setupClientSecret}` to force a clean remount.
- 🟡 `apps/wallet/src/app/api/billing/setup-intent/route.ts:54`, `createSetupIntent` has no idempotency key, so rapid double-clicks (before `addingCard` disables the button) or the race above can create multiple live SetupIntents per user. No financial impact since SetupIntents do not move money and abandoned SetupIntents do not accrue cost; monitor Stripe dashboard clutter if it becomes noticeable.

## Already covered
- Server-side saved-card state (`setDefaultPaymentMethod`) is only ever written from the signature-verified webhook, never trusted from the client -- the client form only drives Stripe's own hosted confirm, so a malicious client cannot forge "card saved" server-side state.
- The SetupIntent is scoped `usage: off_session`, `payment_method_types: ["card"]` server-side, so the client Elements form cannot be tricked into collecting a non-card payment method or requesting a live charge.
- `handleAddCard` and `save` in `autorecharge-panel.tsx` already wrap their fetches in a 10-second `AbortController` timeout plus try/catch/finally, so a hung `/api/billing/setup-intent` request does not leave those specific buttons stuck -- the newly added `AddCardForm.handleSubmit`, flagged above, is the one new path that does not share this protection.
- `POST /api/billing/setup-intent` is both Clerk-authenticated and rate-limited (10 per hour per user) before it will mint a SetupIntent.
