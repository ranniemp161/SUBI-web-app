# Hardening, apps/wallet/src, 2026-07-07

**Analysed by**: systems-level review on Gemini 3.1 Pro (High)
**Scope**: 0 files, uncommitted (Whole Wallet App)
**Risk posture**: Harden before merge

## Summary
The Wallet app's architecture effectively shifts trust from the client to the database, particularly with Stripe checkouts and atomic Postgres operations. However, a significant concurrency race condition exists in the AI Cut charging flow that could lead to double-charging under load, and the IP rate limiter represents an unbounded data growth risk.

## Must-fix before merge
### 🔴 Concurrency & races: Double-charge on AI Cut, `apps/wallet/src/lib/credits.ts:320`
**Scenario**: If a user's browser stutters or they double-click the "Generate AI Cut" button, the frontend could fire two concurrent requests to the backend. Both requests will execute `chargeAiCut` concurrently. Since this method issues an unconditional `UPDATE users SET tokens = tokens - costSeconds` without any idempotency key or lock (unlike `depositPurchase` or `reserveCredits`), the user will be double-charged if they have sufficient balance.
**Impact**: Financial/credits loss for users due to duplicate charges for a single intended AI Cut.
**Mitigation**: Pass a unique idempotency key (e.g., an `idempotencyId` or `jobId` generated by the client) and enforce `ON CONFLICT DO NOTHING` in the `credit_ledger` CTE before updating the `users` balance. Alternatively, enforce a state check on the project row (e.g., `ai_cut_status <> 'processing'`).
**Verify with**: Note: no regression safety net (no test runner configured). Validate via `/verify` with simulated concurrent API calls.

## Should-harden
### 🟠 Scale & performance under load: Unbounded `rate_limits` table growth, `apps/wallet/src/lib/rate-limit.ts:22`
**Scenario**: The IP-based rate limiter inserts a row into `rate_limits` for every unique `x-forwarded-for` IP that hits the webhook. Over time, or under a distributed botnet attack, this table will grow indefinitely because there is no mechanism to evict or `DELETE` expired windows. 
**Impact**: Exhaustion of PostgreSQL storage capacity and degradation of `INSERT ... ON CONFLICT` performance due to massive index bloat.
**Mitigation**: Implement a scheduled background job (e.g., a `pg_cron` extension schedule or a serverless cron endpoint) that runs `DELETE FROM rate_limits WHERE window_start < now() - interval 'X'`.
**Verify with**: Note: no regression safety net. Verify cleanup queries manually.

### 🟠 Adversarial & malformed input (security): Host header spoofing for redirect, `apps/wallet/src/app/api/billing/checkout/route.ts:84`
**Scenario**: The checkout route calculates the `success_url` and `cancel_url` using `process.env.PUBLIC_APP_URL ?? new URL(request.url).origin`. If `PUBLIC_APP_URL` is unconfigured in production, an attacker who proxies a request with a spoofed `Host` header can manipulate `request.url`. Stripe will redirect the user to the attacker's domain after checkout.
**Impact**: Phishing risk; an attacker could capture the user's return flow (and any checkout tokens appended to the URL).
**Mitigation**: Hard-fail or fallback to a relative path if `PUBLIC_APP_URL` is undefined, rather than trusting `request.url`.
**Verify with**: Note: no regression safety net.

## Watch / accept
- 🟡 `apps/wallet/src/lib/credits.ts:221`, `settleHold` clamping to 0 balance if the Deepgram response dictates a massive duration shortfall. Handled gracefully by the system (throwing a Sentry error and clamping) but results in unpaid credits. Acceptable to monitor via Sentry.

## Already covered
- The Stripe webhook uses `ON CONFLICT DO NOTHING` to gracefully survive Stripe retry storms.
- `reserveCredits` and `settleHold` utilize `tokens_hold IS NULL` state gating, completely averting TOCTOU races between duplicate project webhooks.
- Atomic SQL queries prevent the need to read-modify-write balances in memory.
