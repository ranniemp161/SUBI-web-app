# Hardening, main, 2026-07-07

**Analysed by**: systems-level review on Antigravity
**Scope**: 8 files, uncommitted
**Risk posture**: Do not ship

## Summary
The migration from a Postgres-based to an Upstash Redis-based rate limiter introduces two critical production failure modes: a deployment crash due to dropping the Postgres table while active instances still rely on it, and a runtime crash because the Upstash client initialization expects different environment variables than what Vercel KV provides. Furthermore, missing resilience around the Redis calls creates a cascading failure risk during third-party outages.

## Must-fix before merge
### 🔴 Data integrity & migrations, `packages/db/src/schema.ts:157`
**Scenario**: In a standard deployment pipeline, the database migration (`drizzle-kit push` or `migrate`) runs before the serverless functions fully swap to the new release. The migration drops the `rateLimits` table.
**Impact**: The currently running active instances (which still rely on the old Postgres-based rate limit code) will crash with a SQL error (`relation "rate_limits" does not exist`) on every rate-limited endpoint until the new deployment finishes rolling out, causing a hard outage.
**Mitigation**: Revert dropping the `rateLimits` table from `schema.ts` in this PR. Deploy the code changes first to deprecate its usage. In a subsequent release, once no code relies on it, drop the table safely.
**Verify with**: `TESTS = none-by-design` (deployment orchestration check; verify by retaining the `rateLimits` schema).

### 🔴 Configuration initialization mismatch, `apps/wallet/src/lib/rate-limit.ts:19`
**Scenario**: The code explicitly checks for `KV_REST_API_URL` to ensure Vercel KV is configured, but then invokes `Redis.fromEnv()`. The Upstash `fromEnv()` method expects `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN`, not the `KV_` prefixes injected by Vercel KV.
**Impact**: On Vercel, when a rate-limited endpoint is hit, `Redis.fromEnv()` throws an unhandled exception (`Unable to find environment variable: UPSTASH_REDIS_REST_URL`), resulting in an immediate 500 error for all rate-limited endpoints.
**Mitigation**: Explicitly pass the validated Vercel KV environment variables to the constructor: `new Redis({ url: process.env.KV_REST_API_URL!, token: process.env.KV_REST_API_TOKEN! })`.
**Verify with**: `TESTS = none-by-design` (runtime check; verified via `/verify` in production-like environment).

## Should-harden
### 🟠 Security & adversarial input, `apps/wallet/src/lib/rate-limit.ts:45`
**Scenario**: Vercel KV environment variables are accidentally omitted or unlinked from a production project.
**Impact**: The `getRatelimiter` function returns `null`, and the fallback safely but silently returns `allowed: true`. This silently disables rate-limiting in production, opening the system up to abuse or resource exhaustion while hiding a critical misconfiguration.
**Mitigation**: Restrict the `allowed: true` fail-open fallback to development (`process.env.NODE_ENV !== "production"`). In production, missing KV credentials should throw a fatal config error so it fails fast and alerts the team immediately.
**Verify with**: A test validating that missing KV env vars throw or return `allowed: false` when `NODE_ENV === "production"`.

### 🟠 Network & partial failure, `apps/wallet/src/lib/rate-limit.ts:50`
**Scenario**: Upstash Redis (Vercel KV) experiences a network partition, high latency, or a transient outage.
**Impact**: The un-caught `await limiter.limit(key)` throws an exception, bubbling up to Next.js as a 500. This turns a partial failure of a non-critical component (rate limiter) into a full cascading outage of all protected API routes.
**Mitigation**: Wrap the `limiter.limit` call in a `try/catch`. On failure, log the exception via `reportError` and return a fail-open `RateLimitResult` (`allowed: true`) to preserve system availability during the KV outage.
**Verify with**: A mock test in `src/lib/rate-limit.test.ts` verifying rate limiter exceptions are caught and don't bubble.

- 🟡 `apps/wallet/src/app/api/webhooks/stripe/route.ts:77`, Large integers for `tokens` metadata. If a buggy or maliciously spoofed checkout session sets an impossibly high token count exceeding Postgres' 32-bit signed integer limit, the DB insert will throw and cause infinite Stripe retries. Acceptable to monitor since session creation metadata is signed and controlled by our backend.
