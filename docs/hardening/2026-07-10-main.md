# Hardening, main (uncommitted), 2026-07-10

**Analysed by**: systems-level review on gemini-2.5-pro
**Scope**: 20 files, uncommitted vs HEAD
**Risk posture**: Do not ship

## Summary

This change removes the `access_codes` gating system and changes `users.isMember` to default `true`, making the app open-registration. The headline exposure is that **any person with an internet connection can now sign up, create a `users` row, and immediately receive monthly Skool credit grants** — there is no longer any check distinguishing a paying community member from a random signup. The second critical surface is the migration: `DROP TABLE "access_codes" CASCADE` is irreversible and destroys all redemption history including forensic evidence of who was ever granted access. The fallback provisioning path in `getAuthorizedDbUser` now provisions every Clerk-authenticated user unconditionally on first write, compounding the blast radius. The change also passes empty-string email `""` to `provisionUser` from the webhook without a guard, creating permanently un-emailable ghost user rows.

## Must-fix before merge

### Adversarial input / data integrity: empty email persisted as `""`, `apps/rough-cut/src/app/api/webhooks/clerk/route.ts:77-83`
**Scenario**: Clerk sends a `user.created` webhook for an account with no verified email address (possible with social/OAuth providers, or if Clerk's payload shape changes). The fallback chain `emailAddresses?.[0]?.email_address ?? ""` resolves to `""`. Line 83 calls `await provisionUser(userId, "")` with no guard, and the unconditional upsert commits a row with `email = ""` — a `NOT NULL` constraint is satisfied but the value is meaningless.
**Impact**: The user row exists but is silently un-emailable. Any transactional email feature (receipts, credit notifications) dispatches to an empty address or crashes downstream. One bad row per OAuth/social signup that lacks a primary email, permanently until manually corrected.
**Mitigation**: Return a `400` before calling `provisionUser` if `email` is falsy. A `400` causes Svix to retry; a `200` silently confirms the bad row was committed.
**Verify with**: Update the test in `route.test.ts` at line 138 ("uses empty string if no email_addresses are present") — it currently asserts `200` and that `provisionUser` is called with `""`. After the fix it must assert `400` and that `provisionUser` is not called.

### Data integrity / migration: irreversible `DROP TABLE "access_codes" CASCADE`, `packages/db/drizzle/0008_salty_romulus.sql:2`
**Scenario**: The migration runs in production. There is no undo. `DROP TABLE "access_codes" CASCADE` permanently destroys the full redemption history (who redeemed which code, when) and silently drops any FK-dependent objects. If the business decides within hours or days that open registration was a mistake and the gating system must be restored, rollback requires a full point-in-time database restore.
**Impact**: Permanent, irreversible data loss. Forensic audit of who was legitimately granted access is destroyed. No rollback except PITR of the entire database.
**Mitigation**: Before running in production: (1) export the full `access_codes` table to a durable store (S3, cold backup table, or at minimum a CSV attached to the deploy PR); (2) document the data-loss decision in the ADR; (3) include backup confirmation as a required step in the deploy runbook. This is a one-way door — even if open registration is permanent, the backup costs nothing and the data loss is irreversible.
**Verify with**: Operational gate only — no test can prove an irreversible migration is safe. The proof is a confirmed pre-migration backup recorded in the deploy runbook.

### Adversarial input / AuthZ: `getAuthorizedDbUser` provisions every authenticated Clerk user unconditionally with `isMember = true`, `apps/rough-cut/src/lib/authz.ts:21-25` and `apps/wallet/src/lib/authz.ts:21-25`
**Scenario**: Any person completing Clerk email sign-up and hitting any write route causes `getAuthorizedDbUser` to call `provisionUser(clerkId, email)` on first access (before the webhook lands). `provisionUser` upserts a `users` row with `isMember = true` (new schema default), immediately qualifying the user for the monthly Skool credit grant. Previously this fallback required a valid access code. Now it succeeds for every signup — including bot-farm accounts.
**Impact**: Every signup receives full `isMember = true` status and associated monthly credits indefinitely. No remediation path except manual DB updates per account. At scale this is a direct, ongoing financial drain proportional to the number of bot accounts that can be created.
**Mitigation**: If open registration is intentionally permanent, change `isMember` back to `false` as default and grant `true` via a deliberate admin action or a verified Skool membership signal (e.g., a separate webhook from the Skool platform). If `isMember` is being repurposed to mean "any signed-up user," rename the column and update the credit grant job's filter accordingly.
**Verify with**: Add a test to `authz.test.ts` (both apps) asserting the `isMember` value on the provisioned user matches the intended business rule. Currently neither test suite asserts `isMember` on the provisioned row.

## Should-harden

### Time / state: email drift — stored `users.email` diverges from Clerk after a user updates their email, `apps/rough-cut/src/lib/users.ts:17-20`
**Scenario**: The `onConflictDoUpdate` clause uses `set: { email: sql\`${users.email}\` }` — a deliberate no-op that leaves the stored email unchanged on conflict. No `user.updated` Clerk webhook handler exists. Any user who changes their primary email in Clerk causes the stored `users.email` to permanently drift.
**Impact**: Silent misdirected transactional email for any user who updates their Clerk email. Scope grows over time as more users update their email. No automatic correction path.
**Mitigation**: Implement a `user.updated` Clerk webhook handler that updates `users.email` when `email_addresses` changes, or explicitly accept the drift as a known limitation and document it.
**Verify with**: Add a test to `users.test.ts` (both apps) that calls `provisionUser` twice with different emails for the same `clerkId` and asserts the final stored email matches the intended contract.

---

### Resource exhaustion / network: IP rate limit on the webhook risks dropping legitimate events during signup spikes, `apps/rough-cut/src/app/api/webhooks/clerk/route.ts:42`
**Scenario**: Clerk/Svix uses a shared pool of egress IPs. At 120 req/min per IP, a cohort launch or product announcement can cause Clerk's outbound IPs to hit the rate limit. Svix retries with exponential backoff up to 5 times over ~5 hours; if the IP stays rate-limited throughout, the event is permanently dropped. Users who only hit read routes after sign-up never have a `users` row created by either path.
**Impact**: Degraded webhook reliability during signup spikes. Silent lockout for any user whose `user.created` event is permanently dropped and who never triggers the authz fallback (read-only usage pattern).
**Mitigation**: Remove the IP rate limit from the webhook route entirely — the Svix signature is the real security gate. Apply volume ceilings at the Vercel edge/WAF layer where Clerk's known IP ranges can be whitelisted.
**Verify with**: Update `route.test.ts:86-94` to document the deliberate decision (removed limit or explicit risk acceptance).

---

### Observability: `provisionUser` failure in the webhook is unlogged, `apps/rough-cut/src/app/api/webhooks/clerk/route.ts:83`
**Scenario**: If `provisionUser` throws (DB timeout, constraint violation, network partition to Neon), the uncaught exception propagates through `POST`, Next.js returns a 500, and Svix retries — but there is no structured log, no Sentry capture, and no metric on this path. A sustained DB outage during a signup burst results in dozens of silently retrying webhooks with no alert firing.
**Impact**: Silent provisioning failures during DB instability. Users hold valid Clerk sessions but are locked out of every write route until the DB recovers and the retry lands or the authz fallback runs. No alert fires; the team discovers the failure via user complaints.
**Mitigation**: Wrap the `provisionUser` call in a try/catch. On failure: log structured fields (`userId`, error message), capture to Sentry if configured, and return a `500` so Svix retries.
**Verify with**: Add a test to `route.test.ts` where the `provisionUser` mock rejects; assert the response is `500` and that `console.error` (or the Sentry mock) was called with the `userId`.

---

### Data integrity: empty-array destructuring in `provisionUser` silently returns `undefined`, `apps/rough-cut/src/lib/users.ts:14-23`
**Scenario**: `const [user] = await db.insert(...).returning()`. If Drizzle returns an empty array (a DB trigger, RLS policy, or an edge-case zero-row upsert), `user` is `undefined` and `return user` returns `undefined`. Callers type-trust `Promise<User>` and crash with a TypeError on the first property access.
**Impact**: Silent `undefined` propagates into callers instead of a clean thrown error. TypeScript does not catch this — the declared return type hides it. The resulting TypeError surfaces at an unrelated call site and is hard to diagnose.
**Mitigation**: After the upsert, add: `if (!user) throw new Error("provisionUser: DB returned no row for clerkId=" + clerkId)`. This converts a silent `undefined` into a loggable, Svix-retriable error.
**Verify with**: Add a test case to `users.test.ts` (rough-cut) where `returningMock` resolves to `[]` and assert that `provisionUser` throws. The wallet `users.test.ts` already tests the DB-throws case (line 44) but neither file tests the empty-array case.

## Already covered

- **Svix signature verification**: The webhook verifies the signature before any business logic runs. Unsigned or tampered requests are rejected at lines 54-66 with a `400`.
- **`CLERK_WEBHOOK_SECRET` guard**: The route short-circuits with a `500` before touching the rate-limiter or DB if the secret env var is absent (lines 21-29). This prevents silent no-ops in misconfigured environments.
- **Upsert idempotency**: `provisionUser` uses `INSERT ... ON CONFLICT DO UPDATE` — concurrent webhook deliveries and authz-fallback calls for the same `clerkId` are safe; only one row is ever created. The `UNIQUE` constraint on `clerk_id` is the correctness gate, not application logic.
- **`withDbRetry` on the read path**: The DB select in `getAuthorizedDbUser` is wrapped with `withDbRetry`, giving resilience against transient Neon HTTP failures on the most common path.
- **`LIMIT 1` on the user lookup**: Applies `.limit(1)`, preventing a full-table scan if the unique index is somehow unavailable.
- **Clerk session as the authentication gate**: Authorization is two-layered — Clerk session validates identity, `getAuthorizedDbUser` validates that a `users` row exists. A `clerkId` cannot be forged without a valid Clerk session token.
- **Webhook test coverage (post-fix)**: The updated `route.test.ts` correctly mocks `@/lib/ip-rate-limit` (fixing the prior review's broken-mock finding) and covers 500/400/429/signature-fail guard paths.
