# Memory — Security, Hardening & KV Migration

Last updated: 2026-07-07 22:38

## What was built

- **Upload Security**: Implemented a dynamic `maximumSizeInBytes` constraint in `apps/rough-cut/src/app/api/transcribe/blob-token/route.ts` derived from user token balances to prevent duration-spoofing garbage uploads to Vercel Blob.
- **Dashboard Guard**: Added a server-side redirect in `apps/rough-cut/src/app/page.tsx` to automatically route authenticated users away from the landing page directly to `/dashboard`.
- **Production Hardening (KV)**: Fixed `@upstash/redis` instantiation in both `apps/wallet` and `apps/rough-cut` to explicitly use Vercel's `KV_REST_API_URL`. Added a startup crash if KV vars are missing in production (to avoid silent disables), and wrapped `limiter.limit` calls in a `try/catch` to fail-open if the Redis network partitions.
- **Zero-Downtime Rollout**: Reverted a dangerous database migration (`0001_dear_lila_cheney.sql`) that prematurely dropped the `rate_limits` table. The table is restored in `packages/db/src/schema.ts` as `@deprecated` so the old API containers can safely write to it during the Vercel rolling deployment.

## Decisions made

- **Fail-Open Policy**: If the Vercel KV rate limiter goes down, we degrade gracefully by allowing traffic (`allowed: true`) rather than throwing HTTP 500 errors to end users.
- **Rolling Deploy Strategy**: We must always follow a two-phase deployment when dropping tables: Phase 1 deprecates usage but keeps the schema table; Phase 2 (the next PR) actually drops the schema table.

## Problems solved

- Prevented a massive production deployment outage by intercepting the dropped database table before it hit Vercel CI.
- Resolved a configuration mismatch where Upstash Redis threw initialization errors because it defaulted to `UPSTASH_*` environment variables instead of Vercel's `KV_*` format.
